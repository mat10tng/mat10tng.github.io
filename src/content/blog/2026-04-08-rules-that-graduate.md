rules don't start as rules. they start as fixes for specific problems in specific projects. the interesting ones grow into something bigger.

## a real example

in march, I was working on one project and noticed 47 conversations had the same problem — jumping straight into code before aligning on the approach. so that project got a rule: before writing code, state the files you'll change, the approach, and your assumptions. wait for confirmation.

a few weeks later, I started a different project. same pattern showed up. the rule from the first project got referenced, applied, and it worked there too. then a third project. at that point it's clearly not project-specific — it's how the work should always go.

so it graduated from a project lesson to a permanent global rule.

## the promotion path

```
something goes wrong in a project
    → write it down as a lesson
        → the lesson gets referenced in other projects
            → if it keeps being useful across contexts
                → it becomes a permanent rule
```

the system tracks this naturally. every time a note gets referenced in a new conversation, that's a signal. when a lesson keeps getting pulled up across different projects, it's clearly universal. right now there are about 80 permanent rules. most started as one project's fix for one project's problem.

## the opposite direction

some rules that seemed universal turned out to be project-specific. they got referenced in one context but never another. those stay where they are — a rule that helps one project is still a rule worth keeping.

the point is: you don't have to decide upfront whether something is universal. write it down where it happened. if it's universal, the pattern of reuse will make that obvious over time. if it's not, no harm done.

## the principle for your own work

if you keep solving the same kind of problem across different projects, that's a rule waiting to be written down. don't try to predict which lessons will be universal — let the pattern of reuse tell you.

## The Answer
**Stop repeating the same context. Give the model a place to write things down, then make it check that place every time it starts.**

[Want the setup? Jump to Choose Your Path.](#4-choose-your-path)

That is the whole trick. The memory is not magic. It is a folder with session notes, lessons, and a small set of rules that tell the model when to read and write those files.
</div>
</Section>

<Section>
<div slot="insight">
## 1. The Memory Folder
Use one folder for the notes. Give every project a place inside it. Then the model always knows where to look.
</div>

<div slot="technical">
**Folder layout**
```text
memory/
├── state.json (global health)
├── global/
│   └── lessons/ (shared rules)
└── project-x/
    ├── sessions/ (daily logs)
    └── lessons/ (project fixes)
```

**Setup (Linux / macOS)**
```bash
mkdir -p memory/global/lessons
```
</div>
</Section>

<Section>
<div slot="insight">
## 2. Startup Rules
Put the rules in the file your tool reads at startup: `AGENTS.md`, `CLAUDE.md`, `.cursorrules`, or whatever your tool uses.

The rules are simple: read memory before work, write memory after work.
</div>

<div slot="technical">
**Boot rules**
```markdown
# Memory Rules
- On startup, find the current project.
- If `memory/projects/[project-name]` does not exist, run `/init-project`.
- Before big work, read global lessons and this project's lessons.
- If `memory/state.json` says `last_distill` is older than 7 days, run `/distill`.
- At the end of a real session, run `/end-session`.
- The human picks the goal. The LLM keeps the notes up to date.
```
</div>
</Section>

<Section>
<div slot="insight">
## 3. Small Commands
The model only needs three repeatable commands.

**Initialize**: make the folders for a new project.
**Capture**: save what happened at the end.
**Distill**: turn repeated notes into cleaner lessons.
</div>

<div slot="technical">
**Skill: `/init-project`**
> "Scaffold the memory structure: `memory/[project-name]/sessions/` and `memory/[project-name]/lessons/`."

**Skill: `/end-session`**
> "Summarize technical decisions. Save to `memory/[project-name]/sessions/[date]-[goal].md`."

**Skill: `/distill`**
> "Review logs. Move repeated rules to `global/lessons/`. Update `memory/state.json`."
</div>
</Section>

<Section>
<div slot="insight">
## How It Runs
The loop is small:
1. **Boot**: check the memory folder.
2. **Clean**: merge old notes when needed.
3. **Work**: do the task.
4. **Save**: write down what changed.
</div>
</Section>

<Section>
<div slot="insight">
## 4. Choose Your Path
This does not work in a normal chat window.

The model needs somewhere to save files. Pick the path that matches how you use AI.
</div>

<div slot="technical">
**Choose your path**

**Path A: local tool**

1. Download and install a local AI coding tool or terminal-based agent.
2. Create or choose a folder where the memory system should live.
3. Open that folder inside the tool.
4. When the tool asks for permissions, allow file access for that folder.
5. If it asks for terminal or shell permission, allow it.
6. If you want the changes tracked, sign in to git or GitHub inside the tool.
7. Ask the model to create a file called `permission-test.md`.
8. If the file appears, delete it and paste the setup prompt below.

**Path B: web workspace**

1. Use a web AI workspace that has persistent files or projects, not a normal chat window.
2. Create a dedicated project, workspace, or folder for the memory system.
3. If the tool supports it, connect persistent storage such as GitHub, cloud documents, or file export.
4. Ask the model to create a file called `permission-test.md` inside that workspace.
5. Close and reopen the workspace. If the file is still there, paste the setup prompt below.
6. If the file disappears, use the manual version: ask the model to generate the files, then download or copy them into your own memory folder after each session.

The test matters. It proves the model can write somewhere that lasts. If the file disappears when the chat ends, it is not memory yet. It is just chat.
</div>

<div slot="insight">
## 5. The Full Setup Prompt
If I were starting this from zero, I would not ask the model to "remember things better."

I would paste this into a tool that can write files.
</div>

<div slot="technical">
````text
You are running in a tool that can create and edit files.

First, check which kind of tool you are in:

- a local AI coding tool or terminal-based agent
- a web AI workspace with files that stay after the chat closes

Before you begin, verify that you have:

- access to the current project or workspace folder
- permission to create and edit files
- a way to keep those files after this chat ends
- shell/terminal permission if this tool has a terminal
- git access only if this folder is tracked with git

If you cannot create files that stay after the chat ends, stop and tell me this tool is chat-only.
If shell or git access is missing, continue with file edits and note what you could not verify.

Use the file tools you have. Use shell or git only when available.

Your job is to set up a memory folder that works across projects.
The system should let future LLM sessions read context, save what happened,
turn repeated notes into clear lessons, and move lessons that apply to many projects into global lessons.

Do the setup yourself. Do not stop at a plan unless something is genuinely blocked.

Create this memory structure at the workspace root:

memory/
- state.json
- global/
  - lessons/
- projects/
  - [project-name]/
    - sessions/
    - lessons/

Use the current repository, folder, or workspace name as the first project name.

Create or update the project bootloader file used by this coding tool.
Use AGENTS.md, CLAUDE.md, .cursorrules, or the closest equivalent for the environment.
If this web workspace has no startup rules file, create MEMORY_START_HERE.md with the same rules
and tell me where to paste them in the tool's project instructions.

The bootloader must include these rules:

- On startup, identify the current project from the working directory.
- If memory/projects/[project-name] does not exist, initialize it before doing project work.
- Before starting non-trivial work, read global lessons and the current project's recent lessons.
- During work, prefer existing lessons over repeated prompting.
- At the end of every meaningful session, write a session note.
- When the same lesson appears in multiple project notes, move it into memory/global/lessons/.
- Once a week, review old session notes and turn repeated notes into smaller lessons.
- Keep raw sessions searchable, but keep startup context small.
- The human sets goals. The LLM maintains the memory system.

Create three reusable commands or prompt snippets:

1. /init-project
   - Detect the project name.
   - Create memory/projects/[project-name]/sessions/.
   - Create memory/projects/[project-name]/lessons/.
   - Add a short README explaining what belongs there.

2. /end-session
   - Summarize what changed.
   - Capture decisions, bugs fixed, commands that worked, commands that failed, and lessons learned.
   - Save the note to memory/projects/[project-name]/sessions/YYYY-MM-DD-[short-goal].md.
   - If a lesson is reusable, also save it to memory/projects/[project-name]/lessons/.
   - Use the session note template below.
   - Update memory/state.json with last_session_at and session count.

3. /distill
   - Review recent session notes.
   - Remove duplicate or overly specific notes.
   - Merge repeated project lessons into clearer project lessons.
   - Move lessons that apply to many projects into memory/global/lessons/.
   - Use the project lesson and global lesson templates below.
   - Update memory/state.json with last_distill_at.

Initialize memory/state.json with:

- created_at
- last_distill_at
- projects
- global_lesson_count
- session_count

Use these exact markdown templates when writing memory notes.

SESSION NOTE TEMPLATE
Path: memory/projects/[project-name]/sessions/YYYY-MM-DD-short-goal.md

```markdown
---
type: session
project: [project-name]
date: YYYY-MM-DD
goal: "[what we were trying to do]"
outcome: success | struggled | failed | abandoned
reusable: false
---

# [short goal]

## Goal
[One or two sentences describing the work.]

## What Changed
- [User-visible or system-visible change.]
- [Important file, rule, command, or workflow touched.]

## Decisions
- [Decision made and why.]

## Commands
Worked:
- `[command]`

Failed:
- `[command]` - [why it failed or what it taught us]

## Lessons
- [Specific lesson from this session.]

## Follow-Up
- [Optional next action.]
```

PROJECT LESSON TEMPLATE
Path: memory/projects/[project-name]/lessons/short-lesson-title.md

```markdown
---
type: lesson
scope: project
project: [project-name]
created_at: YYYY-MM-DD
source_sessions:
  - memory/projects/[project-name]/sessions/YYYY-MM-DD-short-goal.md
---

# [lesson title]

## Rule
[The reusable rule in one sentence.]

## Why
[The problem this prevents or the pattern it captures.]

## How to Apply
- [When to use it.]
- [What to do differently next time.]

## Boundaries
[When this rule does not apply.]
```

GLOBAL LESSON TEMPLATE
Path: memory/global/lessons/global-rule.md

```markdown
---
type: lesson
scope: global
created_at: YYYY-MM-DD
promoted_from:
  - memory/projects/[project-name]/lessons/short-lesson-title.md
projects_seen:
  - [project-name]
---

# [global rule]

## Rule
[The cross-project rule in one sentence.]

## Why This Became Global
[The repeated signal that made this bigger than one project.]

## How Future Sessions Should Use It
- [Startup behavior, review habit, or implementation constraint.]
- [What the LLM should check before acting.]

## Exceptions
[Cases where the project-local rule should override this.]
```

After setup, run /init-project for the current repository.
Then write one short verification note showing:

- which files you created or updated
- which project name you detected
- how a future session should load context
- what command should be run at the end of a session

Constraints:

- Do not delete or rewrite existing project files unless required.
- Do not store secrets, credentials, private transcripts, or customer data in memory.
- Prefer concise lessons over long logs.
- If a rule is not reusable, keep it project-local.
- If a lesson changes behavior across all projects, put it in global lessons.
````

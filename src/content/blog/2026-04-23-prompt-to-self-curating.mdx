## The Answer
**Stop repeating the same context. Give the model a place to write things down, then make it check that place every time it starts.**

[Want the setup? Jump to Choose Your Path.](#4-choose-your-path)

That is the whole trick. The memory is not magic. It is a folder with session notes, lessons, and a small set of rules that tell the model when to read and write those files.
</div>
</Section>

<Section>
<div slot="insight">
## 1. The Memory Folder
Use one folder for the notes. Give every project a place inside it. Then the model always knows where to look.
</div>

<div slot="technical">
**Folder layout**
```text
memory/
├── state.json (global health)
├── global/
│   └── lessons/ (shared rules)
└── project-x/
    ├── sessions/ (daily logs)
    └── lessons/ (project fixes)
```

**Setup (Linux / macOS)**
```bash
mkdir -p memory/global/lessons
```
</div>
</Section>

<Section>
<div slot="insight">
## 2. Startup Rules
Put the rules in the file your tool reads at startup: `AGENTS.md`, `CLAUDE.md`, `.cursorrules`, or whatever your tool uses.

The rules are simple: read memory before work, write memory after work.
</div>

<div slot="technical">
**Boot rules**
```markdown
# Memory Rules
- On startup, find the current project.
- If `memory/projects/[project-name]` does not exist, run `/init-project`.
- Before big work, read global lessons and this project's lessons.
- If `memory/state.json` says `last_distill` is older than 7 days, run `/distill`.
- At the end of a real session, run `/end-session`.
- The human picks the goal. The LLM keeps the notes up to date.
```
</div>
</Section>

<Section>
<div slot="insight">
## 3. Small Commands
The model only needs three repeatable commands.

**Initialize**: make the folders for a new project.
**Capture**: save what happened at the end.
**Distill**: turn repeated notes into cleaner lessons.
</div>

<div slot="technical">
**Skill: `/init-project`**
> "Scaffold the memory structure: `memory/[project-name]/sessions/` and `memory/[project-name]/lessons/`."

**Skill: `/end-session`**
> "Summarize technical decisions. Save to `memory/[project-name]/sessions/[date]-[goal].md`."

**Skill: `/distill`**
> "Review logs. Move repeated rules to `global/lessons/`. Update `memory/state.json`."
</div>
</Section>

<Section>
<div slot="insight">
## How It Runs
The loop is small:
1. **Boot**: check the memory folder.
2. **Clean**: merge old notes when needed.
3. **Work**: do the task.
4. **Save**: write down what changed.
</div>
</Section>

<div slot="insight">
## 4. Choose Your Path
The memory only works if the notes are saved somewhere.

In a local tool, the model can save them. In a web chat, you save them.
</div>

<div slot="technical">
**Choose your path**

**Path A: local tool**

1. Download and install a local AI coding tool or terminal-based agent.
2. Create or choose a folder where the memory system should live.
3. Open that folder inside the tool.
4. When the tool asks for permissions, allow file access for that folder.
5. If it asks for terminal or shell permission, allow it.
6. If you want the changes tracked, sign in to git or GitHub inside the tool.
7. Ask the model to create a file called `permission-test.md`.
8. If the file appears, delete it and paste the setup prompt below.

**Path B: web chat**

1. Create a folder on your computer, cloud drive, or notes app called `memory`.
2. Paste the setup prompt below.
3. Tell the model you are using web/manual mode.
4. The model should output exact filenames and markdown blocks.
5. Save each block into the matching file.
6. Paste `MEMORY_START_HERE.md` into the tool's project instructions if it has them.
7. At the start of the next chat, paste or attach `MEMORY_START_HERE.md`, `memory/state.json`, and the relevant lesson files.
8. At the end of each chat, ask for a new session note and save it in the path the model gives you.

This version is more manual, but it still works. The model writes the notes. You keep them in the right place.
</div>
</Section>

<div slot="insight">
## 6. After Setup
After the files exist, the system is not something you keep rebuilding.

It becomes a small habit at the start and end of each AI session.
</div>

<div slot="technical">
**If you use a local tool**

1. Open the same project folder.
2. Let the tool read the startup rules and memory files.
3. Tell the model what you want to work on.
4. At the end, ask it to run `/end-session`.
5. The model saves what changed, what worked, and what should carry forward.
6. Every so often, ask it to run `/distill` so repeated notes become cleaner lessons.

**If you use web chat**

1. Start a new chat.
2. Paste or attach `MEMORY_START_HERE.md`, `memory/state.json`, and the relevant lesson files.
3. Tell the model what you want to work on.
4. At the end, ask it for a session note.
5. Save that note into your `memory/projects/[project-name]/sessions/` folder.
6. If the model gives you a reusable lesson, save it in the matching `lessons/` folder.

The local version saves files for you. The web version asks you to save the files. The loop is the same either way: load memory, do the work, save what changed.
</div>

## The Answer
**You can eliminate repetitive prompting and force your technical knowledge to compound across your entire portfolio by establishing an autonomous, project-agnostic memory system.**

Reliability comes from an architecture that performs maintenance by default. By defining structural rules once, you ensure that every session—on any project—initializes, curates, and persists its own engineering standards without manual intervention.
</div>
</Section>

<Section>
<div slot="insight">
## 1. Portfolio Architecture
Compounding requires a strict directory hierarchy. This ensures the model has a deterministic location to store and retrieve history for any project in your portfolio.
</div>

<div slot="technical">
**Unified Memory Structure**
```text
memory/
├── state.json (global health)
├── global/
│   └── lessons/ (portfolio standards)
└── project-x/
    ├── sessions/ (daily logs)
    └── lessons/ (project fixes)
```

**Setup (Linux / macOS)**
```bash
mkdir -p memory/global/lessons
```
</div>
</Section>

<Section>
<div slot="insight">
## 2. System Rules: The Auto-Run Logic
The system is directed by **System Rules** (e.g. `CLAUDE.md`, `.cursorrules`). These are not just instructions; they are the execution logic that forces the model to maintain the architecture autonomously.
</div>

<div slot="technical">
**Mandatory Bootloader Logic**
```markdown
# Project Standards
- **Initialization:** On boot, identify current project. If `memory/[project-name]` does not exist, run `/init-project` immediately.
- **Maintenance:** Check `memory/state.json`. If `last_distill` > 7 days, execute `/distill` before accepting goals.
- **Persistence:** On session completion, autonomously execute `/end-session`.
- **Role:** Architect directs; LLM executes and maintains.
```
</div>
</Section>

<Section>
<div slot="insight">
## 3. Autonomous Skills
The model uses three core skills to maintain the loop without being prompted.

**Initialize**: Scaffolds the environment for new projects.
**Capture**: Persists technical outcomes at the end of work.
**Graduation**: Identifies rules with portfolio-wide utility.
</div>

<div slot="technical">
**Skill: `/init-project`**
> "Scaffold the memory structure: `memory/[project-name]/sessions/` and `memory/[project-name]/lessons/`."

**Skill: `/end-session`**
> "Summarize technical decisions. Save to `memory/[project-name]/sessions/[date]-[goal].md`."

**Skill: `/distill`**
> "Review logs. Identify rules to graduate to `global/lessons/`. Update `memory/state.json`."
</div>
</Section>

<Section>
<div slot="insight">
## The System-Led Workflow
Execution follows a deterministic, autonomous cycle:
1. **Boot**: LLM scaffolds or synchronizes project context.
2. **Curation**: LLM graduates rules if the global state is stale.
3. **Execute**: Architect directs; LLM implements.
4. **Persist**: LLM logs the session automatically before exit.
</div>
</Section>

<div slot="insight">
## 4. The Full Setup Prompt
If I were starting this from zero, I would not ask the model to "remember things better."

I would give it full access, then make the maintenance loop part of its job.
</div>

<div slot="technical">
```text
You have full access to this machine: filesystem, shell, git, and the current working directory.

Your job is to set up a self-curating memory system that works across every project in this portfolio.
The system should let future LLM sessions inherit context, capture what happened, distill durable lessons,
and graduate useful rules from one project to the whole portfolio.

Do the setup yourself. Do not stop at a plan unless something is genuinely blocked.

Create this memory structure at the portfolio root:

memory/
- state.json
- global/
  - lessons/
- projects/
  - [project-name]/
    - sessions/
    - lessons/

Use the current repository folder name as the first project name.

Create or update the project bootloader file used by this coding tool.
Use AGENTS.md, CLAUDE.md, .cursorrules, or the closest equivalent for the environment.

The bootloader must include these rules:

- On startup, identify the current project from the working directory.
- If memory/projects/[project-name] does not exist, initialize it before doing project work.
- Before starting non-trivial work, read global lessons and the current project's recent lessons.
- During work, prefer existing lessons over repeated prompting.
- At the end of every meaningful session, write a session note.
- When the same lesson appears in multiple project notes, graduate it into memory/global/lessons/.
- Once a week, review old session notes and distill them into smaller durable lessons.
- Keep raw sessions searchable, but keep startup context small.
- The human sets goals. The LLM maintains the memory system.

Create three reusable commands or prompt snippets:

1. /init-project
   - Detect the project name.
   - Create memory/projects/[project-name]/sessions/.
   - Create memory/projects/[project-name]/lessons/.
   - Add a short README explaining what belongs there.

2. /end-session
   - Summarize what changed.
   - Capture decisions, bugs fixed, commands that worked, commands that failed, and lessons learned.
   - Save the note to memory/projects/[project-name]/sessions/YYYY-MM-DD-[short-goal].md.
   - If a lesson is reusable, also save it to memory/projects/[project-name]/lessons/.
   - Update memory/state.json with last_session_at and session count.

3. /distill
   - Review recent session notes.
   - Remove duplicate or overly specific notes.
   - Merge repeated project lessons into clearer project lessons.
   - Promote cross-project lessons to memory/global/lessons/.
   - Update memory/state.json with last_distill_at.

Initialize memory/state.json with:

- created_at
- last_distill_at
- projects
- global_lesson_count
- session_count

After setup, run /init-project for the current repository.
Then write one short verification note showing:

- which files you created or updated
- which project name you detected
- how a future session should load context
- what command should be run at the end of a session

Constraints:

- Do not delete or rewrite existing project files unless required.
- Do not store secrets, credentials, private transcripts, or customer data in memory.
- Prefer concise lessons over long logs.
- If a rule is not reusable, keep it project-local.
- If a lesson changes behavior across all projects, put it in global lessons.
```

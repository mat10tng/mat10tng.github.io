## Principle
Prompting is a fallback. Reliability is achieved by moving from adjective-heavy instructions to a persistent **system** that surrounds the model. 

## Technical Requirements
To move beyond brittle prompts, the environment must satisfy three functional requirements:
1. **Persistence**: Context must survive across independent work sessions.
2. **Retrieval**: The model must prioritize local project data over its general training.
3. **Self-Curation**: The system must automatically identify and remember lessons from your work.
</div>

<div slot="technical">
| Requirement | Implementation |
|---|---|
| **Persistence** | Local Memory directory |
| **Retrieval** | Mandatory System Rules |
| **Curation** | Graduation of session logs |
</div>
</Section>

<Section>
<div slot="insight">
## 1. Memory: The Foundation
Memory provides the physical structure for short-term activity and long-term engineering standards. It satisfies the requirement of **Persistence** via a strict directory hierarchy.
</div>

<div slot="technical">
**Folder Structure**
```text
memory/
├── state.json (global metadata)
├── global/
│   └── lessons/ (universal)
└── project-x/
    ├── sessions/ (activity logs)
    └── lessons/ (local standards)
```

**Linux / macOS**
```bash
mkdir -p memory/global/lessons memory/project-x/{sessions,lessons}
```
</div>
</Section>

<Section>
<div slot="insight">
## 2. The Skills
Before the system can run, the model must be taught how to interact with Memory. These "skills" are reusable prompt instructions.

### Capture (`/end-session`)
Summarizes the current session and writes it to the correct project directory.

### Self-Curation (`/distill`)
**Self-curation** is the core of the system. It is the process of identifying recurring engineering patterns in raw session logs and promoting them to permanent rules in the `lessons/` directory.
</div>

<div slot="technical">
**Skill: `/init-project`**
> "Scaffold the Memory structure for a new project: `memory/name/{sessions,lessons}`."

**Skill: `/end-session`**
> "Summarize technical decisions. Save to `memory/project-x/sessions/[date]-[goal].md`."

**Skill: `/distill`**
> "Identify 3 recurring patterns from recent logs. Move to `global/lessons/`. Update `memory/state.json`."
</div>
</Section>

<Section>
<div slot="insight">
## 3. System Rules
System Rules define the operating environment inside the **LLM**. A root file (e.g. `CLAUDE.md`, `.cursorrules`) establishes the Architect/LLM relationship and enforces the maintenance schedule.
</div>

<div slot="technical">
**Project Root Configuration**
```markdown
# Project Standards
- **Init:** On boot, read `memory/state.json`.
- **Curation:** If `last_distill` > 7 days, execute `/distill`.
- **Role:** Architect directs; LLM executes.
- **Log:** Use `/end-session` to persist results.
- **Skills:** [/init-project, /end-session, /distill]
```
</div>
</Section>

<Section>
<div slot="insight">
## The Daily Workflow
The result is an evolving system that starts with context. The LLM inherits its environment automatically, and the architecture absorbs new standards through a verified pattern of reuse.

1. **Boot**: LLM reads root state and context.
2. **Curation**: LLM graduates rules if stale.
3. **Execute**: Architect directs; LLM implements.
4. **Capture**: Run `/end-session` to persist the log.
</div>

<Section>
<div slot="insight">
A recurring pattern in system failure: when a component consistently fails a task, the problem is often an incorrect assignment of responsibility. 

Reliability is often achieved by removing the task from the AI entirely and transferring it to a deterministic part of the system.
</div>
</Section>

<Section>
<div slot="insight">
## The Primary Question

When a system fails, the question is not "How can the agent do this better?" but "Should the agent be doing this at all?"

Deterministic tasks require 100% precision. Humans and AI are both probabilistic at enforcement. Structural logic should be encoded, not prompted.
</div>

<div slot="technical">
| Failure Type | Correct Owner |
|---|---|
| **Bookkeeping** | System/DB |
| **Enforcement** | CI/Scripts |
| **Logic Branches** | Code |
| **Context Synthesis** | Agent |
</div>
</Section>

<Section>
<div slot="insight">
## Implementation Example

Consider the creation of duplicate IDs in a document graph. Instead of instructing the agent to "be careful," implement a backend validation.
</div>

<div slot="technical">
```javascript
// Deterministic Validation Logic
function validateIdUniqueness(newId, existingIds) {
    if (existingIds.has(newId)) {
        throw new Error(`Duplicate ID detected: ${newId}`);
    }
}
```
</div>
</Section>

<Section>
<div slot="insight">
## Verification

Success is measured by the cessation of the specific failure pattern. If a backend service handles the logic, the agent can no longer "forget" to execute it. The system becomes simpler and more reliable than one that relies on continuous prompting.

## Technical Insight

When a component keeps failing, before attempting optimization, verify whether that component should own the responsibility. Transferring the job is more effective than refining the instruction.
</div>
</Section>

The monthly health check ran on april 22. Compounding score: **86%**.

I've been building a knowledge base system for three weeks — capturing session notes, distilling lessons, feeding prior work back into new sessions. The theory was that it would compound. Sessions would start smarter over time because they'd have context instead of starting from zero. But I didn't have numbers until now.

86% means 773 out of around 900 sessions successfully reused prior knowledge. They found something relevant in the base before starting. Three weeks ago that number was 0% — the base was empty.

## what 86% means in practice

The compounding score isn't "percentage of sessions where something useful happened." It's more specific: the percentage of sessions where the system already had something relevant *before* they started — prior work the session could build on rather than rediscover.

The 64% → 86% jump in three weeks came from two things: more notes to find, and better tagging so search could actually surface them. The knowledge base grew from 488 to 903 documents. Capture completeness went from 0% to 44% — about half of sessions now leave a trace the next session can use.

The number that surprised me most wasn't the score. It was 773 vs 131 — sessions with reuse vs without. The system is no longer the exception. It's the baseline.

## what the measurement missed

Same week, different kind of data. The daily codebase sweep on April 23 flagged a file with an unused import — `import asyncio`, nothing in the file body using it. Standard cleanup, removed it.

Two new test failures.

The import was being re-exported somewhere down the dependency chain. The linter couldn't see it. The automated sweep couldn't see it. Only running the full test suite after the change caught it.

That's the thing about automated measurement: it tells you what it's designed to look for. The compounding score tells me sessions are reusing knowledge — it doesn't tell me which 14% aren't, or why. The daily sweep tells me tests pass — it doesn't tell me about hidden dependencies in code that looks clean.

Good metrics tell you direction. They don't tell you what you're not measuring.

<div slot="insight">

## if you're building something like this

Watch the trend more than the number. 0% → 44% capture completeness is more interesting than 44% itself. The trend tells you whether the system is self-reinforcing — whether adding sessions makes future sessions better, or whether you're just filling a database nobody reads.

The jump from 64% → 86% in three weeks is the number I was waiting for. Not because 86% is impressive on its own. Because the direction is right, and I have something to measure against next month.
